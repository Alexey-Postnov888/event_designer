# Пользовательские требования

**1\. Требования администратора (создателя мероприятий)**

**1.1. Доступ и аутентификация**

1.  Администратор должен иметь возможность входа в систему через личный аккаунт с логином и паролем.
2.  Система должна позволять восстановление доступа через код, отправляемый на электронную почту.
3.  После входа администратор должен быть перенаправлен на панель управления с перечнем своих мероприятий.

**1.2. Управление мероприятиями**

1.  Администратор должен иметь возможность создавать новое мероприятие.
2.  Администратор должен иметь возможность редактировать созданное мероприятие.
3.  Администратор должен иметь возможность удалять мероприятие.
4.  Система должна сохранять статус мероприятия (например, «Черновик» или «Активно»).
5.  Администратор должен видеть статус каждого мероприятия.

**1.3. Контент мероприятия**

1.  Администратор должен иметь возможность добавлять название, описание и обложку мероприятия.
2.  Администратор должен иметь возможность загружать карту площадки (спрайт, изображение или схема).
3.  Администратор должен иметь возможность добавлять на карту интерактивные точки.
4.  Для каждой точки администратор должен иметь возможность добавлять описание, изображение или ссылку.
5.  Администратор должен иметь возможность редактировать и удалять точки.
6.  Администратор должен иметь возможность добавлять расписание мероприятия (таймлайн).
7.  Администратор должен иметь возможность связывать события таймлайна с точками на карте.
8.  Все внесённые изменения должны сохраняться автоматически или вручную по команде пользователя.

**1.4. Предпросмотр и публикация**

1.  Администратор должен иметь возможность просматривать мероприятие перед публикацией.
2.  Предпросмотр должен отображать карту, точки и таймлайн в том же виде, что и финальный лендинг.
3.  Администратор должен иметь возможность опубликовать мероприятие.
4.  После публикации система должна предоставить публичную ссылку на мероприятие.
5.  Опубликованное мероприятие должно быть доступно пользователям в мобильном приложении.
6.  Администратор должен иметь возможность снять мероприятие с публикации или скрыть его из публичного доступа.

**1.5. Мониторинг и уведомления (обсудим на дейлике)**

1.  Администратор должен иметь возможность получать уведомление о завершении публикации мероприятия.
2.  Администратор должен иметь возможность просматривать статистику просмотров мероприятия.
3.  Администратор должен иметь возможность получать уведомления о технических ошибках при публикации или загрузке файлов.

**2\. Требования пользователя (участника мероприятия)**

**2.1. Авторизация и доступ**

1.  Пользователь должен иметь возможность авторизоваться через одноразовый код, отправленный на электронную почту.
2.  Пользователь должен иметь возможность получить доступ только к опубликованным мероприятиям.
3.  Пользователь должен иметь возможность выйти из системы.

**2.2. Просмотр мероприятий**

1.  Пользователь должен видеть список доступных мероприятий.
2.  Пользователь должен иметь возможность открыть мероприятие и просмотреть карту, интерактивные точки и таймлайн.
3.  Пользователь должен иметь возможность кликать по интерактивным точкам и получать дополнительную информацию (описание, фото).
4.  Пользователь должен иметь возможность переходить по элементам таймлайна.
5.  Пользователь должен иметь возможность просматривать мероприятие в офлайн-режиме после загрузки.

**2.3. Уведомления**

1.  Пользователь должен получать push-уведомления о приближении событий из расписания.
2.  Пользователь должен иметь возможность включать или отключать уведомления.

**3\. Общие пользовательские требования**

**3.1. Интерфейс и удобство**

1.  Интерфейс конструктора должен быть понятен пользователю без изучения документации.
2.  Все ключевые действия (создание, редактирование, публикация) должны быть доступны не более чем в 3 клика.
3.  Интерфейс должен поддерживать работу в браузерах Яндекс, Chrome.
4.  Интерфейс должен быть адаптирован под экраны ноутбуков и мониторов.
5.  Мобильное приложение должно быть совместимо с Android 8.0 и выше.
6.  Система должна иметь единый визуальный стиль в фирменных цветах Т-Банка.

**3.2. Надёжность и устойчивость**

1.  В случае потери соединения веб-конструктор должен сохранять несохранённые изменения в браузере.
2.  Мобильное приложение должно продолжать работу в офлайн-режиме.
3.  При восстановлении соединения данные должны синхронизироваться с сервером.
4.  Все пользовательские данные должны храниться безопасно и не теряться при обновлении версии системы.

**3.3. Безопасность и доступ**

1.  Все действия пользователей должны требовать авторизации.
2.  Пароли и коды авторизации не должны храниться в открытом виде.
3.  Доступ к данным должен быть разграничен по ролям (администратор / пользователь).
4.  Передача данных должна осуществляться только по защищённому протоколу HTTPS.

**4\. Требования к качеству взаимодействия**

1.  Система должна реагировать на пользовательские действия без задержек более 2 секунд.
2.  Ошибки системы должны сопровождаться понятным сообщением пользователю.
3.  После успешного выполнения операции пользователь должен получать визуальное подтверждение.
4.  Все основные функции должны быть доступны независимо от скорости сети (в пределах разумных ограничений).
